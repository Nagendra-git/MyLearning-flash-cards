Domain Name System (DNS)

DNS is a distributed and hierarchical naming system that translates human-readable domain names into IP addresses.

    Example:
        google.com ->142.250.xxx.xxx
    Instead of remembering IP addresses, users type domain names.

Why Do We Need DNS?

    Imagine every website required users to remember its IP address.
    This would be immposible to maintain.
    DNS acts like the Internet's phone book.

Why Can't We Use Domain Names Directly?

    Computers communicate using IP addresses.
    DNS performs this translation.

DNS Lookup Flow

    Suppose the user enters:
        https://www.google.com
    The browser performs the following steps:
    Browser -> Local DNS Cache -> OS Cache ->Router Cache ->ISP DNS Resolver ->
    Root DNS Server ->TLD Server (.com) -> Authoritative DNS Server ->IP Address
    ->Browser connects to Google

Step-by-Step DNS Resolution

    Step 1: Browser Cache:
        The browser first checks whether it already knows the IP.
        Browser Cache
            google.com -> 142.250.xxx.xxx
        If found, no network request is needed.
    
    Step 2: Operating System Cache:
        If the browser doesn't have it:
            chrome ->Windows DNS Cache
        The operating system may already know the answer.

    Step 3: Router Cache:
        Some home or office routers cache DNS responses.
        pc -> wi-fi router -> Cached Ip.
    
    Step 4: ISP DNS Resolver (Recursive Resolver)
        If not found anywhere:
            Browser -> ISP DNS Resolver
        Examples:
            Google Public DNS (8.8.8.8)
            Cloudflare (1.1.1.1)
        The resolver now finds the answer on behalf of the client. 
    Recursive Resolver  
            A Recursive Resolver performs the entire lookup process and returns the final IP to the client.
        Client -> Recursive Resolver -> Internet DNS Hierarchy -> IP Address
        The client makes only one request.
    step 5: Root DNS Server
        The resolver doesn't know where google.com is.
            It first contacts a Root DNS Server.
        It does not return the IP.
    step 6: Top-Level Domain (TLD) Server
            Now the resolver asks:
                .com Server ->Where is google.com?
            The TLD server replies:
            Ask Google's authoritative server.
            Again, it doesn't return the IP.
    step 7 :Authoritative DNS Server
            Finally:
                Resolver -> Google Authoritative DNS    
            It responds:google.com ->142.250.xxx.xxx
            Now the resolver returns this IP to the browser.

What is an Authoritative DNS Server?

    This server stores the official DNS records for a domain.
        Example:
        google.com ->A Record ->142.250.xxx.xxx
    It provides the definitive answer for that domain.

DNS Records

    1. A Record (Most Important) -> Maps a domain to an IPv4 address(142.250.xxx.xxx)
    2. AAAA Record ->Maps a domain to an IPv6 address.(2404:6800:4009::200e)
    3. CNAME Record ->Creates an alias for another domain.
                        ww.google.com (google.com)
                        Useful when multiple names point to the same service.
    4.MX Record ->Specifies the mail server. (Used for email delivery.)
                    google.com ->Mail Srever
    5. NS Record ->Specifies the authoritative name servers for the domain.
                    example.com -> ns1.example.com
    6. TXT Record ->Stores arbitrary text.
                        Common uses:
                                SPF
                                DKIM
                                Domain verification

What is DNS Caching?

    Once an IP is found:
        google.com -> 142.250.xxx.xxx
    It is cached so future lookups are faster.  
    Benefits:
        Faster response times
        Reduced DNS traffic
        Lower latency
        Reduced load on authoritative servers

What is TTL (Time To Live)?

    Every DNS record has a TTL.
        Example:TTL = 300 seconds   
        Meaning:
            Cache this result
            for 5 minutes.
        After expiration, a fresh DNS lookup is performed.

Summary Table

Concept Key Point
DNS Translates domain names to IP addresses
Recursive Resolver Performs the complete lookup for the client
Root Server Directs queries to the appropriate TLD server
TLD Server Directs queries to the correct authoritative server
Authoritative Server Returns the official DNS record
Browser Cache First place checked for cached DNS entries
OS Cache System-level DNS cache
TTL Defines how long a DNS record remains cached
A Record Domain → IPv4 address
AAAA Record Domain → IPv6 address
CNAME Domain alias → another domain
MX Mail server information
NS Authoritative name servers
TXT Verification and email security metadata
DNS Caching Improves performance and reduces server load
DNS Round Robin Basic load balancing by rotating returned IPs 
    🎯 Interview Takeaway:
        DNS is hierarchical: Client → Recursive Resolver → Root → TLD → Authoritative Server.
        Caching and TTL are critical for performance and reducing lookup traffic.
        Recursive resolvers hide the complexity of the lookup process from clients.
        Authoritative servers provide the final, trusted answer.
        DNS is highly distributed to ensure scalability, fault tolerance, and high availability.
        DNS-based load balancing is simple but limited because it lacks real-time health and load awareness, 
        so modern systems often combine DNS with dedicated load balancers.
