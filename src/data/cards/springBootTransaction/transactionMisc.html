Spring Proxy

    Who starts the transaction?
        Not your method.
        Spring creates another object called a Proxy.
        Your class knows nothing about transactions.
        The Proxy manages everything.
    InternalFlow :
        bankService.transferMoney();
    Actually becomes
        Proxy.transferMoney()
            ↓
        transactionManager.begin()
            ↓
        realObject.transferMoney()
            ↓
        transactionManager.commit()

Why Proxy?

    Because Spring uses AOP (Aspect-Oriented Programming).
    The Proxy adds additional behavior
    without modifying your code.
    Examples
        Logging
        Security
        Transactions
        Caching

3. Self Invocation Problem

    Problem:
        @Service
        public class EmployeeService {
            @Transactional
            public void saveEmployee() {
                saveAudit();
            }
            @Transactional(
            propagation = Propagation.REQUIRES_NEW
            )
            public void saveAudit() {

            }
        }
        Note :- does not go through Spring Proxy.

        solution 1 :- move to another service.
        solution 2 :-Inject the proxied bean into itself 
        (less common and generally discouraged because it complicates the design):
        @Service
        public class OrderService {
            @Autowired
            private OrderService self;
            public void placeOrder() {
                self.payment();
            }
            @Transactional
            public void payment() {
            }
        }

Misc

    private methods(No) ->Because Spring Proxy cannot intercept private methods.
    Why doesn't self invocation work? -> bypasses the Spring Proxy.
    Who actually starts the transaction? -> PlatformTransactionManager.
    static methods ->Static methods are not invoked on a proxied object instance,
                     so Spring's proxy-based transaction advice cannot intercept them.

What is a CGLIB Proxy?

   
    CGLIB (Code Generation Library) is a library that creates a subclass of your class at runtime.
    Spring uses this subclass (proxy) to add extra behavior like:
            Transactions (@Transactional)
            Logging
            Security
            Caching

    class BankService {
        public void transferMoney() {
            System.out.println("Transfer");
        }
    }
    class BankServiceProxy extends BankService {
        @Override
        public void transferMoney() {
            beginTransaction();
            super.transferMoney();
            commitTransaction();
        }
    }
    Note: This is a simplified illustration. 
    CGLIB generates the subclass dynamically at runtime; 
    it doesn't literally create this source code.

    When Does Spring Use CGLIB?
Situation	                            Proxy Used
Bean implements an interface	        JDK Dynamic Proxy (by default)
Bean has no interface	                CGLIB Proxy
Class-based proxy explicitly requested	CGLIB Proxy

In modern Spring Boot applications, 
CGLIB is commonly used because class-based proxies are often enabled by default.
