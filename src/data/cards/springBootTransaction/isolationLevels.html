Isolation Levels

    Spring supports four Isolation levels.
    @Transactional(isolation = Isolation.READ_COMMITTED)

1.READ_UNCOMMITTED

    Allows
        Dirty Read ✅
        Non-repeatable Read ✅
        Phantom Read ✅

    Nothing is prevented.
    Example:
    T1              T2
    update salary   read updated salary
    (Not commited)  Allowed

    Use case:
    Very Rarely
    Performance is high.
    Consistancy is lowest.

2. READ_COMMITTED

    Prevents:
        Dirty Read ❌
    Allows
        Non-repeatable Read ✅
        Phantom Read ✅
        This is the default isolation level in databases such as PostgreSQL and Oracle.
    Example:
    T1              T2
    update salary   read updated salary
    (Not commited)  Not Allowed
    Only after commited.

3. REPEATABLE_READ

    Prevents:
        Dirty Read ❌
        Non-repeatable Read ❌
    Allows
        Phantom Read ✅
        This is the default isolation level for MySQL InnoDB.
    Example:
    T1              T2
    Read Balance    Update Balance
    5000            commit
    read again
    5000
    The same row remains consistent throughout T1.

4. SERIALIZABLE

    Highest isolation.
        Prevents
            Dirty Read ❌
            Non-repeatable Read ❌
            Phantom Read ❌
            Transactions execute as if they were one after another.
    Example:
    T1              T2
    select *        Cann't Insert
    From Employee   Wait....
    Until T1 finishes.
    Performance is slower.
    Consistency is highest.

Summary Table

Isolation Level Dirty Read Non-repeatable Read Phantom Read
READ_UNCOMMITTED ✅ Allowed ✅ Allowed ✅ Allowed
READ_COMMITTED ❌ Prevented ✅ Allowed ✅ Allowed
REPEATABLE_READ ❌ Prevented ❌ Prevented ✅ Allowed*
SERIALIZABLE ❌ Prevented ❌ Prevented ❌ Prevented

* Phantom reads are prevented in some databases (e.g., InnoDB in MySQL) using next-key locking, but according to the SQL standard they are allowed under REPEATABLE_READ.
