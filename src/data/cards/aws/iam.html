Identity and Access Management(IAM)

One of the most important services within AWS is IAM.
It controls who can access AWS resources and what actions they can perform.

What is IAM?
IAM allows you to:
    Create users
    Create groups
    Create roles
    Attach permissions
    Secure AWS accounts
    Enable MFA
    Provide temporary credentials
    Federate identities
    Control access using policies

It follows the Principle of Least Privilege
    Give only the permissions that are absolutely necessary.

IAM is Global

IAM is a Global Service.
Unlike EC2 or RDS, IAM does not belong to a specific region.
AWS Account
    |
    |------ IAM (Global)
               |
               |---- Users
               |---- Groups
               |---- Roles
               |---- Policies

Create a user once.
That user can access resources in any AWS Region (depending on permissions).

IAM Policies

Policies define permissions.
Policies are JSON documents.
Example
{
    "Version":"2012-10-17",
    "Statement":[
        {
            "Effect":"Allow",
            "Action":"s3:ListBucket",
            "Resource":"*"
        }
    ]
}

Policy Structure

Version -Current version(2012-10-17)
Statement
Effect - allow/Deny
Action
Resource
Condition
Example
"Condition": {
  "IpAddress": {
      "aws:SourceIp":"10.10.0.0/16"
  }
}
Only users from the office network can access.

Policy Evaluation Logic
AWS evaluates all applicable policies together.
Default
    Everything is Denied
Need explicit
    Allow

Explicit Deny Always Wins:
Example
    Allow S3
    Allow EC2
    Deny Delete Bucket
Delete bucket
    ❌ Denied
Even though another policy allows it.

Evaluation Order

↓
Explicit Alow
↓
Explicit Deny (Highest Priority)

Allows centralized access to multiple AWS accounts and applications.

Multi-Factor Authentication (MFA)

Adds a second authentication factor beyond the password.
Common methods:
    Authenticator apps
    Hardware security keys
    FIDO2/WebAuthn devices
Example:
    Username
    Password
    +
    6-digit OTP

Best practice:
    Enable MFA for the root user immediately.
    Enable MFA for privileged IAM users.
Access Keys
    Used for programmatic access.
Consist of:
    Access Key ID
    Secret Access Key
Best practices:
    Never hardcode keys in source code.
    Rotate keys regularly.
    Prefer IAM roles over long-term access keys when possible.

ARN (Amazon Resource Name)

Every AWS resource has a unique ARN.
Example:
    arn:aws:s3:::my-bucket
    arn:aws:iam::123456789012:user/Nagendra
    arn:aws:ec2:us-east-1:123456789012:instance/i-1234567890abcdef0
General format:
    arn:partition:service:region:account-id:resource

Security Best Practices

1.Never use the root user for daily work.
2.Enable MFA for the root user and administrators.
3.Grant least privilege permissions.
4.Prefer IAM roles over access keys.
5.Rotate credentials regularly.
6.Remove unused users and permissions.
7.Use groups instead of assigning permissions individually.
8.Use customer-managed policies for reusable permission sets.
9.Review IAM Access Analyzer findings periodically.
10.Audit accounts using IAM Credential Reports.
11.Avoid using "Resource": "*" unless necessary.
12.Avoid using "Action": "*" unless full administrative access is required.
