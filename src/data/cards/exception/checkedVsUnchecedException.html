Checked vs Unchecked Exceptions

    Java divides exceptions into two categories.
    Throwable
        │
        ├── Error
        │
        └── Exception
                │
                ├── Checked Exceptions
                │      ├── IOException
                │      ├── SQLException
                │      └── ClassNotFoundException
                │
                └── RuntimeException (Unchecked)
                        ├── NullPointerException
                        ├── ArithmeticException
                        ├── IllegalArgumentException
                        ├── NumberFormatException
                        └── IndexOutOfBoundsException

Compile-Time vs Runtime

Checked Exception Unchecked Exception
Checked by compiler Checked by JVM at runtime
Must handle or declare No need to handle explicitly
Represents recoverable conditions Represents programming bugs
Extends Exception (excluding RuntimeException) Extends RuntimeException

Why are RuntimeExceptions Unchecked?

    Runtime exceptions usually indicate programming mistakes.
    String s = null;
    s.length();
    int a = 10 / 0;
    int[] arr = new int[5];
    arr[10] = 100;
    These are coding errors that should be fixed, not forced to be caught.
    Imagine if Java forced this:
        try {
            s.length();
        }
        catch(NullPointerException e){
        }
    Every line of code would need a try-catch, making programs unreadable.
    Therefore, Java leaves it to the developer to write correct code instead of forcing exception handling.

Why are Checked Exceptions Checked?

    Checked exceptions usually represent external problems beyond your control.
        Examples:
            File not found
            Database connection lost
            Network timeout
            Printer unavailable
        Example:
            FileReader reader = new FileReader("data.txt");
            Can the compiler know if the file exists?
            No.
            So Java forces you to decide:
            Handle it Or propagate it

Common Checked Exceptions

    | Exception              | When it occurs           |
    | ---------------------- | ------------------------ |
    | IOException            | File operations          |
    | FileNotFoundException  | File missing             |
    | SQLException           | Database access          |
    | ClassNotFoundException | Reflection/Class loading |
    | InterruptedException   | Thread interruption      |

Common Unchecked Exceptions

    | Exception                      | Cause                     |
    | ------------------------------ | ------------------------- |
    | NullPointerException           | Accessing a null object   |
    | ArithmeticException            | Divide by zero            |
    | IllegalArgumentException       | Invalid method argument   |
    | NumberFormatException          | Invalid number conversion |
    | ArrayIndexOutOfBoundsException | Invalid array index       |
    | IndexOutOfBoundsException      | Invalid collection index  |
    | ClassCastException             | Invalid type casting      |

Real-World Examples

    Checked Exception
        Open Bank Statement PDF -> File Missing -> IOException
        The user may choose another file or retry.
        The application can recover.
    Unchecked Exception
        Developer writes
        account.getBalance() -> account is null ->NullPointerException
        This is a programming bug.
        The user cannot fix it.

When Should We Use Checked Exceptions?

    Use Checked Exceptions when:
        The caller can recover.
        The caller should decide what to do.
    Examples:
        Reading a file
        Calling a database
        Calling an external API
        Sending an email
        Reading configuration files
    Example:
        public void uploadFile() throws IOException
        The caller can retry or show an error message.

When Should We Use Runtime Exceptions?

    Use Runtime Exceptions when:
        The error is caused by incorrect code.
        Recovery isn't expected.
        It represents a business rule violation.
    Examples:
        throw new IllegalArgumentException("Age cannot be negative");
                    or
        throw new UserNotFoundException("User not found");
    🎯 30-Second Interview Answer
        Java exceptions are classified into checked and unchecked exceptions. 
        Checked exceptions are verified by the compiler and must be handled or 
        declared because they usually represent recoverable conditions like file or database failures. 
        Unchecked exceptions extend RuntimeException and represent programming errors or business rule violations, 
        such as NullPointerException or IllegalArgumentException. 
        In Spring Boot, custom runtime exceptions are commonly used with a global exception handler 
        to produce clean and consistent API responses.
    💡 Memory Trick
        Checked  = Outside my control → Recover → Handle
                    (File, DB, Network)
        Unchecked = My coding mistake or business rule → Fix Code
                    (Null, Divide by Zero, Invalid Input)
