Error Handling in Go

Go encourages developers to return and handle errors explicitly instead of throwing exceptions.

Go Error Handling Flow

Function
   │
   ▼
Returns (value, error)
   │
   ▼
Check if err != nil
   │
   ├── Yes → Handle error
   │
   └── No → Continue execution

Errors

An error represents something that went wrong during program execution.
In Go, an error is simply a value that implements the built-in error interface.

type error interface {
    Error() string
}

Any type that implements Error() string satisfies the error interface.
Why Return nil?
When no error occurs:
return result, nil
nil means: No error occurred.

Creating Errors:
   Using errors.New()
   
import "errors"
err := errors.New("Invalid username")
   
   Using fmt.Errorf(): 
   
err := fmt.Errorf("employee %d not found", id)
   
   Use fmt.Errorf() when you need formatted error messages.

   Custom Errors:
   
type ValidationError struct{
   Message string
}
func (e ValidationError) Error() string{
   return e.Message
}
   
   Now:
   
return ValidationError{
   Message : "Salary cannot be negative";
}
   
   Since it implements Error() string, it satisfies the error interface.

   Error Wrapping:
      Introduced in Go 1.13.
      Instead of:
      
return fmt.Errorf("database error")
      
      Wrap the original error:
      
return fmt.Errorf("saving employee: %w", err)
      
      %w preserves the original error while adding more context.

   errors.Is():
      Checks whether an error matches another error, even if it's wrapped.
      
if errors.Is(err, sql.ErrNoRows) {
   fmt.Println("Employee not found")
}
      
      This is the preferred way to compare wrapped errors.

   errors.As():
      Used when you need to extract a specific custom error type.
      
var validationErr ValidationError
if errors.As(err, &validationErr){
   fmt.Println(validationErr.Message)
}
      
      Use errors.As() for types, and errors.Is() for specific error values.
